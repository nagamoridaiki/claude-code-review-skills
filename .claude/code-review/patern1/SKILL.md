## レビュー対象

以下のコマンドで変更差分を取得し、レビューしてください。

### 1. main ブランチからの差分（commit 済みを含む）

現在のブランチが `main` でない場合、ブランチ上の変更をレビュー対象とします。

```bash
git diff main...HEAD
```

`main...HEAD` は分岐点以降にこのブランチで加えた変更のみを対象とします（PR レビューと同等）。

### 2. 未コミットの変更

作業ツリー上の未コミット変更も合わせてレビューしてください。

```bash
git diff
git diff --cached
```

### 3. 未追跡ファイル

未追跡の新規ファイルがある場合は、ファイル内容を直接読んでレビューしてください。

```bash
git status --short
```

## レビュー観点

1. **バグ・ロジックエラー** — 意図しない動作、エッジケースの見落とし、off-by-one エラー
2. **セキュリティ** — インジェクション、ハードコードされた秘密情報、安全でない入力処理
3. **可読性・保守性** — 不明瞭な命名、不必要な複雑さ、過剰な抽象化
4. **一貫性** — 既存コードベースのスタイル・パターンとの整合性

## 言語固有のチェックポイント

変更されたファイルの言語に応じて、`languages/` ディレクトリ内の該当ファイルを参照してください:
- TypeScript/JavaScript → `languages/typescript.md`

各ファイルには、言語特有のよくあるミスと参考資料がまとめられています。

## 出力言語

**レビュー結果はすべて日本語で出力してください。** タイトル、要約、指摘、観察事項、ステータス報告を含め、英語は使用しないでください。

## 出力フォーマット

問題がある場合のみ報告してください。問題がなければ「問題なし」と報告してください。

レビュー全体の構成例:

```
## コードレビュー: <変更の概要>

**ステータス:** 問題なし / 要修正あり

### 変更概要
（変更内容の要約）

### 指摘事項
（問題がある場合のみ）

### 所見
（良い点・気になった点など）
```

各指摘は以下の形式で簡潔に:

```
[重要度: high/medium/low] ファイルパス:行番号
内容: 問題の説明
提案: 修正案
```
