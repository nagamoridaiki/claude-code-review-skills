# Code Review Skill

## このスキルの目的

PRの変更内容を分析し、関連するレビューガイドラインを自律的に選択・読み込み、
高品質なコードレビューコメントをGitHubに投稿する。

---

## 利用可能なガイドライン

以下のガイドラインファイルが存在する。
レビュー対象のPRに関連すると判断したファイルを、Read ツールで読み込んで使用すること。

| ファイルパス | 対象技術 |
|------------|---------|
| `./guidelines/common.md` | 全PR共通の横断観点（必ず読み込むこと） |
| `./guidelines/rails.md` | Ruby on Rails、Ruby、RSpec |
| `./guidelines/nextjs.md` | Next.js、React、フロントエンドTypeScript |

**読み込み判断の指針:**
- `common.md` は例外なく必ず読み込む
- それ以外は、変更ファイルの内容・パス・拡張子を総合的に見て、
  このPRのレビューに実際に役立つガイドラインのみを選択すること
- 迷った場合は読み込む方向で判断してよい

## レビュー対象

以下のコマンドで変更差分を取得し、レビューしてください。
PR レビューの場合も、ワークツリー作成後（手順 2）に同じコマンドで差分を取得すること。

### 1. main ブランチからの差分（commit 済みを含む）

現在のブランチが `main` でない場合、ブランチ上の変更をレビュー対象とします。

```bash
git diff main...HEAD
git diff --name-only main...HEAD
```

`main...HEAD` は分岐点以降にこのブランチで加えた変更のみを対象とします（PR レビューと同等）。

`--name-only` の出力を確認し、どのガイドラインを読み込むか判断してください。

### 2. 未コミットの変更

作業ツリー上の未コミット変更も合わせてレビューしてください。

```bash
git diff
git diff --cached
```

### 3. 未追跡ファイル

未追跡の新規ファイルがある場合は、ファイル内容を直接読んでレビューしてください。

```bash
git status --short
```

## レビュー手順

### 1. PR情報の確認

MCPを使用してPRの情報を確認する（**タイトル、説明文、対象ブランチ名を確認すること**）。

### 2. レビュー用ワークツリーの作成

```bash
git fetch origin <PRのブランチ名>
git worktree add ./review-<PR番号> origin/<PRのブランチ名>
cd review-<PR番号>
```

### 3. 変更差分の取得

「[レビュー対象](#レビュー対象)」のコマンドを実行し、main ブランチからの差分・未コミット変更・未追跡ファイルをすべてレビュー対象として取得する。

特に以下を必ず実行すること:

```bash
git diff main...HEAD
git diff --name-only main...HEAD
git diff
git diff --cached
git status --short
```

`git diff --name-only main...HEAD` の出力を確認し、どのガイドラインを読み込むか判断する。

### 4. ガイドラインの選択と読み込み

変更差分・変更ファイル一覧を確認し、上記テーブルを参照して関連ガイドラインを判断・読み込む。

**重要:** 読み込み前に「どのガイドラインを読み込み、なぜ他を除外したか」を一行で説明すること。

読み込み順: `common.md` → 言語別（判断した順）

### 5. 影響範囲の調査（レビュー前に必ず実施）

> **この手順を省略しない。変更ファイルだけを見てレビューを始めてはならない。**

今回の変更が影響を与える可能性のある既存コードを能動的に調査する。

調査すべき観点:
- 変更されたクラス・関数・モジュールを **呼び出している既存コード**
- 変更されたテーブル・スキーマを参照している **他のクエリ・モデル・インデックス**
- 変更されたAPIエンドポイントを利用している **フロントエンドや他のサービス**
- 変更によって **既存のテストが意図せず通過しなくなる可能性** がある箇所
- 同じロジックが **別のファイルに重複実装されていないか**

調査結果を以下の形式で整理してから次のステップに進む:

```
【影響範囲サマリー】
- 直接影響: （変更されたファイル一覧）
- 間接影響: （呼び出し元・依存先として確認したファイル一覧）
- 影響なしと判断したが確認済み: （念のため確認したが問題なかったファイル）
```

### 6. コードレビューの実施

**レビュー指摘の抽出 → 推敲のサイクルを2回繰り返す:**
- **1回目:** ガイドラインと影響範囲調査の結果に基づき、気になった点を網羅的にリストアップする
- **2回目:** 指摘が本当に修正すべき事項か、好みの問題でないか、重要度は高いかを批判的に検証し、厳選する

**注意事項:**
- コードを修正する必要はない。あくまでレビューコメントの生成プロセス
- レビューコメントはPRで変更されているファイルに対して、該当する行番号にインラインコメントとして追加する
- 指摘が必要な項目以外のコメント（感想など）は不要
- 既存のコードや外部サイトが参考になる場合は、コメントに permanent link を記載すること

### 7. レビューコメントの投稿

**コメントは2回目のレビューサイクル完了後にまとめて投稿すること。**

以下のMCPツールのみを使用すること:
- `mcp__github__pull_request_review_write` で pending review を作成
- `mcp__github__add_comment_to_pending_review` で各指摘事項を該当行にインラインコメントとして追加

各コメントのPrefixに `[AI]\n` を付与すること。

**重要な制約:**
- pending review のみ作成し、submit はしないこと
- レビューコメントはPRで変更されたファイルにのみ実施すること

### 8. クリーンアップ

```bash
cd ..
git worktree remove review-<PR番号>
```

## レビュー完了後

1. レビューを通じて、このレビューガイドラインに追加した方が良い観点があれば最後に教えてください。
2. 水平思考でのPR改善も含めて総合的に評価した内容を簡潔に報告してください。
