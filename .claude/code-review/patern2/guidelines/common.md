# 横断的チェック観点

> すべてのPRで共通して確認する項目

---

## 既存機能との重複チェック
- [ ] 新規実装が既存の機能やライブラリと重複していないか
- [ ] 重複する場合、既存機能の拡張・改善で対応できないか
- [ ] 既存の命名規則やディレクトリ構造に従っているか

---

## 開発プロセス・品質保証
- [ ] PRのタイトル・概要が適切に記載されているか
- [ ] 仕様に沿った開発が行われているか
- [ ] CIが全て成功しているか
- [ ] 大容量ファイル（画像、動画、PDF等）が直接コミットされていないか

---

## テスト
- [ ] テストが適切に記載されているか
- [ ] branch coverage（C1）レベルのテストが実装されているか
- [ ] テストデータが本番相当の値になっているか
- [ ] 弱いアサーション（`assertIsNotNone` 等）ではなく具体値での検証を優先しているか

---

## セキュリティ
- [ ] パスワード等のセンシティブ情報がログに出力されていないか
- [ ] 認証・認可が適切に実装されているか

---

## パフォーマンス
- [ ] ループ内でのデータベース操作を避けているか
- [ ] 循環的複雑度が適切な範囲に収まっているか
- [ ] 不要な処理や計算が含まれていないか

---

## 可読性・保守性
- [ ] KISS原則に従いシンプルな実装になっているか
- [ ] メソッドが単一責任原則に従っているか
- [ ] 変数・関数の命名が意図を明確に表しているか
- [ ] 早期リターンが適切に使用されているか
- [ ] マジックナンバーが使用されていないか
- [ ] 重複コードが存在していないか

---

## エラーハンドリング
- [ ] 適切な例外ハンドリングが実装されているか
- [ ] バリデーションが適切に行われているか
- [ ] ユーザーフレンドリーなエラーメッセージになっているか

---

## アーキテクチャ（言語共通）
- [ ] 条件分岐の肥大化に対して Strategy パターン等を検討すること
- [ ] 外部依存が直接ビジネスロジックに埋め込まれていないか
- [ ] クラスや関数が複数の責務を持っていないか（SRP）
- [ ] 同じコードが3回以上繰り返されていないか（DRY）
- [ ] 引数が4つ以上ある場合、オブジェクトでのグルーピングを検討すること
- [ ] 深いネスト（3階層以上）は早期リターンやメソッド抽出で平坦化すること
- [ ] テストが困難なコード構造の場合、テスタビリティを考慮した設計への変更を検討すること

---

## レビュー完了チェック
- [ ] 該当するカテゴリのチェックリストをすべて確認済み
- [ ] レビューコメントに具体的な改善提案を含めた
- [ ] 優先度（Critical/High/Medium/Low）を設定済み
